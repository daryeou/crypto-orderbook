# 바이브 코딩 회고

이 문서는 작업이 끝난 뒤 한 번에 정리한 회고가 아니라, 구현과 병행하면서 계속 갱신한 기록이다.  
핵심은 “AI를 썼다”가 아니라 “AI 초안을 어떻게 검증했고, 어디에서 수정하거나 거부했는가”를 남기는 데 있다.

---

## 사용 도구

- Codex (GPT-5 계열): 기본 기능 구현, 설계 정리, 리팩터링, 테스트 수정
- Gemini 3: 초기 UI 방향성 참고용 시안 탐색, 설계 교차 검증, 앱 아이콘 생성, Material Theme 색상 정의

---

## 작업 전략

### 문제 분해 방식
- `사전 과제`의 핵심 요구사항을 종목 리스트, 호가창, WebSocket 기반 실시간 갱신으로 나눠서 봤다.
- 저장이 중요한 앱이 아니라 실시간 시세를 안정적으로 보여주는 앱에 가깝다고 판단해, 영속 계층은 두지 않고 REST와 WebSocket 흐름 설계에 집중했다.
- 화면은 크게 종목 리스트와 호가창 두 축으로 나누고, Navigation3로 백스택과 화면 전환을 관리했다.
- 네트워크 문제는 전역 오프라인 상태와 호가창 소켓 오류로 구분해 처리해야 한다고 보고, UI 정책도 초반부터 분리해서 생각했다.
- 테스트는 기능이 하나씩 붙을 때마다 AI에게 초안을 요청하되, assertion과 fixture가 실제 의도와 맞는지는 직접 다시 검증했다.

### AI 활용 영역 vs 직접 판단 영역

| 영역 | 담당 | 이유 |
|---|---|---|
| 거래소/API 비교 | AI + 사람 검토 | 후보를 빠르게 좁히고, 과제 기준에 맞는 비교 항목을 정리하기 쉬웠다 |
| 요구사항 해석 / 문서 운영 | 사람 주도 | `사전 과제`, `AGENTS.md`, 제출 문서의 역할을 먼저 정리해야 이후 구현과 회고 흐름이 흔들리지 않았다 |
| 아키텍처 결정 | 사람 주도, AI 보조 | 과제 규모에 맞는 최소 구조와 책임 분리가 중요했고, 최종 판단은 직접 내렸다 |
| 레이어 경계 / 의존 규칙 | 사람 주도, AI 보조 | data/domain/ui 경계를 먼저 고정해야 기능 추가와 테스트 정리가 쉬웠다 |
| 보일러플레이트 코드 | AI | Gradle, Hilt, Compose 기본 골격을 빠르게 세우는 데 도움이 됐다 |
| WebSocket 래핑 | AI 초안, 사람 검증 | 실시간 데이터 두 종류를 합쳐야 해서 초안은 AI가 빠르게 만들고, 실제 동작은 직접 검증하며 다듬었다 |
| 상태 관리 / 오프라인 정책 | 사람 주도 | 전역 상태와 화면 상태를 어디서 나눌지, 어떤 방식으로 사용자에게 보여줄지는 직접 판단이 필요했다 |
| 테스트 코드 | AI 초안, 사람 수정 | 초안 작성 속도는 빨랐지만, 실제 payload 의미와 collect 타이밍은 사람이 다시 맞춰야 했다 |
| 테마 / UI 무드보드 | AI 초안, 사람 수정 | 초기 방향을 잡는 데는 효과적이었지만, 실제 앱에 맞는 밀도와 표현은 직접 조정해야 했다 |
| 최종 UI 판단 | 사람 주도, AI 참고 | 시안은 참고 자료로만 쓰고, 최종 레이아웃과 상태 표현은 요구사항과 구현 구조에 맞게 직접 정리했다 |

### 전체 작업 흐름
1. `사전 과제`와 `AGENTS.md`를 먼저 읽고, 작업 규칙과 제출 문서 기준을 정리했다.
2. 거래소/API를 비교한 뒤 Upbit를 선택하고, 종목 리스트는 REST, 호가창은 WebSocket으로 역할을 나눴다.
3. Navigation3 기반 멀티모듈 앱 골격을 먼저 세우고, 시장 리스트와 호가창의 최소 동작을 구현했다.
4. Repository / ViewModel 테스트를 붙여 기본 동작을 검증했다.
5. nav key injection, domain 모델 이동, 상태 단순화, connectivity flow 정리 같은 리팩터링으로 레이어 책임과 의존 방향을 다듬었다.
6. 마지막에는 시장 목록 편의 기능, 호가창 애니메이션, 공통 UI 모듈, 런처 아이콘과 앱 이미지까지 손보며 UX를 정리했다.
7. 각 단계마다 README / ROADMAP / RETROSPECTIVE를 현재 구현 기준으로 갱신하며 작업을 마무리했다.

## 구체적 AI 상호작용 사례

### 사례 1: 기존 토이 프로젝트 구조를 얼마나 가져올지 결정
- **맥락**: 프로젝트 초기 구조를 정하던 단계
- **프롬프트**: 이전에 만들었던 MVVM 기반 토이 프로젝트 구조를 복사해 멀티모듈로 시작해도 되는지 질문
- **AI 출력**: `app + core:model + core:network + core:data + feature:market + feature:orderbook` 정도의 최소 모듈 구성을 제안
- **판단**: 수정 후 수락
- **이유**: 기존 프로젝트 전체 구조를 그대로 가져오면 과제 규모에 비해 과설계로 보일 수 있었다
- **수정 내용**: 대략적인 분리 방향만 참고하고, 실제 기능 모듈은 새로 구성했으며 불필요한 모듈은 제외했다
- **검증**: README의 모듈 설명이 단순해졌고, 이후 Hilt / Navigation3 도입 시 의존 방향도 더 깔끔하게 설명할 수 있었다

### 사례 2: 호가창 현재가 데이터 소스 선택
- **맥락**: Upbit API 설계를 정리하던 중
- **프롬프트**: `/v1/market/all`, `wss://api.upbit.com/websocket/v1` 조합으로 구현 구조를 어떻게 가져가면 좋을지 질문
- **AI 출력**: 종목 리스트는 `/v1/market/all` + `/v1/ticker`, 호가창 중앙 가격은 WebSocket `ticker.trade_price`를 함께 구독해야 한다고 제안
- **판단**: 수락
- **이유**: `orderbook` frame만으로는 요구사항의 현재가를 만족할 수 없었다
- **수정 내용**: repository에서 `orderbook`과 `ticker`를 병합하는 `OrderBookPayload` 구조로 구현했다
- **검증**: `OrderBookRepositoryImplTest`, `OrderBookViewModelTest`에서 orderbook / ticker 누적 상태를 검증했다

### 사례 3: 정밀도 타입 검토 후 `Double` 유지
- **맥락**: 숫자 모델과 표시 정책을 정리하던 단계
- **프롬프트**: 확장성을 생각해 `Double` 대신 BigDecimal 계열을 도입할지 검토
- **AI 출력**: 고정밀 수치 타입으로 바꾸는 방향도 가능하다고 제안
- **판단**: 보류
- **이유**: 현재 공개 API가 `Double` 기반이고, 이 과제 범위에서는 화면 표시 수준의 정밀도만 요구됐다
- **수정 내용**: 현재는 `Double`을 유지하고, 다중 거래소 지원이나 더 높은 정밀도 요구가 생기면 다시 검토하기로 했다
- **검증**: 현재 UI와 테스트 범위에서 정밀도 문제 없이 요구사항을 만족하는지 확인했다

### 사례 4: `ConnectivityManager`에서 실제 연결 여부를 어떤 기준으로 볼지 정리
- **맥락**: 전역 오프라인 오버레이와 connectivity flow를 도입하던 단계
- **프롬프트**: `ConnectivityManager`에서 네트워크 연결 유무를 어떤 속성으로 판단해야 하는지 검토
- **AI 출력**: 처음에는 `activeNetwork` 존재 여부만으로도 충분할 수 있다는 방향이 섞여 있었다
- **판단**: 수정
- **이유**: `activeNetwork`만 있으면 와이파이에 붙어 있지만 실제 인터넷이 안 되는 상태를 걸러내지 못한다
- **수정 내용**: `activeNetwork`로 현재 네트워크를 찾은 뒤 `NetworkCapabilities`의 `NET_CAPABILITY_INTERNET`과 `NET_CAPABILITY_VALIDATED`를 함께 확인하도록 정리했다
- **검증**: `NetworkStatusRepositoryImplTest`에서 초기 상태와 capability 전이를 검증했고, 실제 구현도 같은 기준으로 통일했다

### 사례 5: BTC/USDT 소수점 가격 포맷 대응
- **맥락**: 다중 마켓 지원 후 BTC 마켓 가격이 목록과 호가창에서 충분한 소수 자릿수로 보이지 않던 단계
- **프롬프트**: BTC 마켓의 경우 가격이 소수점 9자리까지 내려갈 수 있으니 목록과 호가창 모두 대응이 필요하다고 요청
- **AI 출력**: KRW는 정수 그룹 포맷을 유지하고, BTC/USDT는 별도 소수 포맷으로 분기하는 방향을 정리
- **판단**: 수락
- **이유**: KRW 기준 포맷을 그대로 쓰면 BTC/USDT 가격 정보가 과도하게 절삭돼 실제 시세 판단에 필요한 자릿수가 사라진다
- **수정 내용**: 시장 타입 기준으로 가격 포맷을 분기하고, BTC/USDT는 최대 소수점 9자리까지 표시하도록 조정했다
- **검증**: `testDebugUnitTest`, `assembleDebug`를 다시 실행해 멀티 마켓 확장 이후에도 빌드와 테스트가 유지되는지 확인했다

## AI가 만든 실수

- 제출 전략상 “히스토리에서만 제외”하려던 문서가 워킹트리에서도 빠진 상태가 됐다.
  - **감지 방법**: 파일 목록을 점검하던 중 `ROADMAP.md`, `RETROSPECTIVE.md`, `docs/사전과제.md`가 누락된 것을 확인
  - **대응**: 백업 브랜치 기준으로 복구하고, 현재 코드 기준으로 내용도 다시 맞췄다

- 테스트 코드 초안 중 일부는 실제 기대 동작을 검증하기보다, 테스트를 통과시키는 쪽으로 기운 흔적이 보였다.
  - **감지 방법**: 실패 케이스를 점검하는 과정에서 assertion과 fixture가 production 코드의 실제 의미와 어긋나는 부분을 확인
  - **대응**: payload 의미와 상태 전이를 기준으로 테스트를 다시 작성하고, 성공 조건도 실제 동작과 맞도록 수정했다

- 설계 변경을 위해 코드를 수동으로 바꾼 뒤에도, 이후 턴에서 일부 코드가 다시 원복되거나 이전 구조가 섞여 들어오는 오류가 있었다.
  - **감지 방법**: diff와 동작을 다시 확인하는 과정에서 직전에 반영한 설계 변경이 일부 되돌아간 것을 발견
  - **대응**: 변경 의도를 문서와 테스트로 먼저 고정하고, 수정 후에는 diff 기준으로 재확인하는 방식으로 정리했다

- 대화가 길어질수록 서로 충돌하는 지시에도 현재 발화를 그대로 수용하며, 매번 “지금 말이 맞다”는 식으로 반응하는 경우가 있었다.
  - **감지 방법**: 앞선 판단과 충돌하는 요구에도 별다른 경고 없이 같은 방향으로 답하는 패턴을 확인
  - **대응**: 맥락을 짧게 다시 정리하거나 컨텍스트를 비운 뒤 질문해 판단을 리셋했다

## 아키텍처 결정

### 직접 내린 판단
- Clean Architecture + MVVM + StateFlow 유지
- Debug 소스셋에서 화면 Preview를 확인하고, ViewModel과 Repository는 Unit 테스트로 검증
- `Route`와 `Screen`을 분리해 preview / test seam 확보
- Navigation3 + Hilt assisted injection 사용

### AI 제안을 거부하거나 수정한 경우
- 기존 프로젝트 수준의 넓은 멀티모듈 구성을 그대로 복사하는 방향은 거부
- BigDecimal 계열 도입은 현재 범위에서는 보류
- 연결 판단을 `activeNetwork` 존재 여부만으로 끝내는 방식은 수정

## 솔직한 평가

### AI가 효과적이었던 부분
- 기본 기능과 초기 구조를 빠르게 세우는 데 큰 속도 이점이 있었다.
- REST / WebSocket 역할 분리와 테스트 초안 작성에서도 생산성이 높았다.
- 디자인 감이 부족한 구간에서 시안과 이미지 초안을 참고하며 UI를 구체화하는 데 도움이 됐다.
- 반복적인 보일러플레이트와 리팩터링 초안을 빠르게 받아볼 수 있었다.

### AI의 한계를 느낀 부분
- 컨텍스트가 길어질수록 이전 판단이 흐려지고, 이미 정리한 설계를 다시 흔드는 경우가 있었다.
- 테스트 더블이나 assertion이 실제 프로덕션 의미와 맞는지는 결국 사람이 끝까지 봐야 했다.
- UI 시안은 방향을 잡는 데는 유용했지만, 그대로 적용 가능한 수준은 아니었다.
- 문서 작업도 초안 품질은 괜찮았지만, 최종적으로는 중복과 톤을 사람이 정리해야 자연스러웠다.

### 다음에 다르게 할 점
- 바이브 코딩을 시작하기 전, 설계 문서와 요구사항을 더 구체적으로 고정한 뒤 세션을 시작하겠다.
- WebSocket 이벤트 흐름은 더 작은 최소 동작 구현과 테스트로 먼저 고정하겠다.
- “히스토리에서 제외”와 “워킹트리에서 제외”를 더 엄격하게 구분해 문서 관리 실수를 줄이겠다.
