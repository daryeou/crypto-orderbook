# AGENTS.md

이 문서는 이 저장소에서 작업하는 **AI 코딩 에이전트**(Claude Code, Cursor, Copilot 등)를 위한 지침입니다. 사람 개발자는 참고만 해도 됩니다.

이 과제는 "AI 도구 활용 과정"이 평가 항목이며, 산출물 중 `RETROSPECTIVE.md`, `README.md`, `ROADMAP.md` 세 문서를 **작업과 함께 지속적으로** 갱신해야 합니다. 세션이 끝난 뒤 몰아서 쓰면 사례가 유실되고 품질이 떨어집니다.

---

## 0. 과제 컨텍스트 (필독)

- **목표**: Android(Kotlin + Jetpack Compose)로 암호화폐 거래소 종목 리스트 + 실시간 호가창 구현
- **Must-have**: (1) REST로 종목 리스트, (2) 호가창(매수/매도 + 현재가), (3) **WebSocket 실시간 갱신** (REST polling 금지)
- **평가 비중**: 구현 및 검증 프로세스 35% > WebSocket 25% > UI/UX·가점 20% = 코드 품질·아키텍처 20%
- **제출물**: 동작하는 앱 + `README.md` + `ROADMAP.md` + `RETROSPECTIVE.md` + 커밋 히스토리(squash 금지)

자세한 원문은 저장소 루트의 `docs/사전과제.md` 참고.

## 1. 문서 갱신 규칙 (가장 중요)

작업 단위가 끝날 때마다 아래 체크리스트를 돌립니다. "작업 단위"는 대략 **커밋 1개 또는 기능 1개**입니다.

### 1-1. RETROSPECTIVE.md — AI 상호작용 로그

다음 중 하나에 해당하면 **즉시** `RETROSPECTIVE.md`의 "구체적 AI 상호작용 사례" 섹션에 항목을 추가합니다:

- AI에게 코드/설계/디버깅을 요청했고 결과물이 커밋에 반영됨
- AI 제안을 **거부**하거나 **크게 수정**함 (이 케이스가 특히 중요 — 판단력을 보여주는 근거)
- AI가 명백한 실수를 했고 사람이 발견함 → "AI가 만든 실수" 섹션에도 추가

각 사례는 아래 형식을 따릅니다:

```markdown
### 사례 N: (제목)
- **맥락**: 어떤 작업 중이었는지 1줄
- **프롬프트**: 요약 또는 원문
- **AI 출력**: 요약 (코드면 핵심만)
- **판단**: 수락 / 수정 / 거부
- **이유**: 판단 근거 (기술적, 설계적, 성능적 등)
- **수정 내용**: 수정했다면 무엇을 어떻게
```

**최소 3개, 권장 5개**. 사소한 자동완성은 제외하고, 설계 결정이 걸린 것 위주로 기록합니다.

### 1-2. README.md — 결정 사항 반영

다음이 확정되거나 변경되면 README의 해당 섹션을 갱신:

| 트리거 | 갱신 섹션 |
|---|---|
| 거래소 선택/변경 | "거래소 선택 근거" |
| 라이브러리 추가/교체 | "주요 라이브러리" + 선택 근거 |
| 아키텍처 레이어 구조 변경 | "아키텍처" + 다이어그램 |
| 빌드/실행 방법 변경 | "빌드 & 실행" |
| 요구사항 모호점 해석 | "가정과 판단" |

### 1-3. ROADMAP.md — 미구현/보류 추적

다음 상황에서 ROADMAP에 항목 추가:

- Nice-to-have를 의도적으로 스킵 (에러 처리, 테스트 등)
- 시간 부족으로 일부만 구현 (예: 재연결은 되는데 백오프는 미구현)
- 리팩토링 대상 발견 (TODO 주석만 달지 말고 ROADMAP에도 기록)
- 테스트를 작성하지 않았다면 **반드시 테스트 전략 섹션 채우기** (과제 요구사항)

## 2. 커밋 규칙

- **squash 금지** (과제 명시)
- 커밋 단위는 작고 의미 있게. "WIP", "fix" 같은 뭉뚱그린 메시지 지양
- 컨벤션 제안: `feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:`
- AI가 작성한 코드라도 **사람 이름**으로 커밋 (과제는 최종 판단 주체를 봄)
- 문서 갱신은 관련 기능 커밋에 포함하거나 직후 별도 `docs:` 커밋으로

## 3. 기술 가이드라인 (판단 시 기본값)

아래는 **기본값**이며 더 나은 근거가 있으면 바꾸고 README/RETROSPECTIVE에 이유를 남깁니다.

- **언어/UI**: Kotlin + Jetpack Compose (필수)
- **거래소 후보**: Upbit (한국, WebSocket orderbook 지원, 한국 IP에서 테스트 쉬움) / Bithumb / Binance
  - 선택 시 WebSocket 호가 스트림 지원 여부 반드시 확인
- **아키텍처 후보**: Clean Architecture(data/domain/ui) + MVI 또는 MVVM + StateFlow
  - 이 규모에선 과한 레이어링 금지. 3~4개 레이어 이내로.
- **비동기**: Coroutines + Flow (WebSocket은 `callbackFlow`로 감싸기)
- **네트워크**: Retrofit(REST) + OkHttp WebSocket 또는 Ktor Client
- **DI**: Hilt (Koin도 허용, 근거 남기기)
- **직렬화**: kotlinx.serialization
- **이미지**: Coil (필요 시)
- **테스트**: JUnit5 + Turbine(Flow) + MockK

**WebSocket 처리 체크리스트** (평가 25%):
- [ ] `callbackFlow` 또는 유사 패턴으로 Flow 래핑
- [ ] 라이프사이클 연동 (화면 벗어나면 구독 해제)
- [ ] 재연결 로직 (백오프 권장)
- [ ] 연결 상태 UI 노출 (로딩/에러)
- [ ] 백프레셔 고려 (호가는 초당 수십 건 들어올 수 있음 — `conflate()`, `sample()` 검토)

## 4. 작업 흐름 권장 순서

평가 항목 "구현 및 검증 프로세스(35%)"가 가장 큰 비중이므로, 순서와 검증을 문서에 남기는 것이 중요합니다.

1. **거래소 API 스펙 확인** → README "거래소 선택 근거" 작성
2. **최소 동작 스파이크**: REST 1개 호출 + WebSocket 1개 구독이 성공하는 더미 앱
3. **도메인 모델 설계** → 레이어 구조 README에 그림
4. **종목 리스트 (Must-have 1)**
5. **호가창 화면 + WebSocket 연동 (Must-have 2, 3)**  ← 여기가 핵심
6. **에러/로딩/빈 상태 처리** (Nice-to-have)
7. **테스트 최소 1~2개** (WebSocket Flow 변환 로직 중심 권장)
8. **README/ROADMAP/RETROSPECTIVE 최종 점검**

각 단계에서 막히면 **막힌 지점 자체**를 RETROSPECTIVE에 기록하세요 — 평가자는 문제 해결 과정을 봅니다.
